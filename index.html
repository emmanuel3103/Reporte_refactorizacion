Inyección de Dependencias

Implementación en API Node.js — Proyecto EDI 301

Aldahir Ballina | Marzo 2026

1. Introducción

El proyecto EDI 301 es una aplicación de gestión de familias escolares compuesta por un frontend desarrollado en Flutter y un backend construido con Node.js, Express y MSSQL. A lo largo de su desarrollo, el código fue evolucionando hacia una arquitectura limpia con separación en capas: infraestructura, casos de uso, controladores y rutas. Sin embargo, a pesar de esta separación estructural, persistía un problema de acoplamiento fuerte entre las capas: cada módulo importaba directamente sus dependencias mediante require(), lo que dificultaba las pruebas unitarias, la sustitución de implementaciones y el mantenimiento a largo plazo. Este documento describe la implementación del principio de Inyección de Dependencias (DI) aplicado al backend Node.js del proyecto EDI 301, identificando el problema original, justificando el patrón elegido, mostrando el código antes y después de la refactorización, y analizando los beneficios obtenidos.

2. Descripción del Problema de Acoplamiento

2.1 Estructura antes de la refactorización

Antes de aplicar DI (Inyección de Dependencias), el código seguía un patrón de importación directa en cada capa. Los controladores importaban directamente los casos de uso, y los casos de uso importaban directamente la infraestructura:

auth.controller.js (ANTES)

// El controlador crea su propia dependencia (acoplamiento fuerte)
const AuthUseCase = require('../../../application/use-cases/auth/auth.usecase');

exports.login = async (req, res) => {
  const user = await AuthUseCase.login({ login, password });
};

auth.usecase.js (ANTES)

// El caso de uso importa infraestructura directamente
const { sql, queryP } = require('../../../infrastructure/database/db');
const { comparePassword } = require('../../../shared/utils/hash');

async function login({ login, password }) {
  const rows = await queryP(Q.byLogin, { Login: { type: sql.NVarChar, value: login } });
}

2.2 Problemas identificados

ProblemaDescripción
Acoplamiento fuerteCada módulo crea o importa directamente sus dependencias.
Dificultad para pruebasNo se puede usar mock sin modificar código.
Violación del DIPDependencia de implementaciones concretas.
Baja cohesiónMezcla lógica con creación de dependencias.

3. Solución Aplicada — Inyección de Dependencias

3.1 Patrón elegido

Constructor Injection + Contenedor Manual sin frameworks externos.

3.2 Contenedor DI

class Container {
  constructor() {
    this._factories  = new Map();
    this._singletons = new Map();
  }

  register(name, factory, singleton = true) {
    this._factories.set(name, { factory, singleton });
    return this;
  }

  resolve(name) {
    const { factory, singleton } = this._factories.get(name);
    if (singleton) {
      if (!this._singletons.has(name)) {
        this._singletons.set(name, factory(this));
      }
      return this._singletons.get(name);
    }
    return factory(this);
  }

  get(name) { return this.resolve(name); }
}

3.3 Bootstrap

const container = require('./container');

container.register('db', () => require('./database/db'));

container.register('authUseCase', (c) =>
  new AuthUseCase(c.get('db'))
);

container.register('authController', (c) =>
  new AuthController(c.get('authUseCase'))
);

4. Código Refactorizado

ANTES

const { sql, queryP } = require('...db');

DESPUÉS

class AuthUseCase {
  constructor(db) {
    this.queryP = db.queryP;
  }
}

5. Análisis de Beneficios

BeneficioDescripción

Ciclo de vidaSingleton controlado.
ExtensibilidadEscalable.

6. Conclusiones

La implementación de DI permitió reducir el acoplamiento sin frameworks externos. Constructor Injection fue la mejor opción por simplicidad y control.
